## Summary
The cc10x plugin's `hooks.json` mixes 6 real/current hook event names (`PreToolUse`, `SessionStart`, `PostToolUse`, `SubagentStop`, `PreCompact`, `Stop`) with **4 non-standard/invented event names** (`TaskCompleted`, `PostCompact`, `StopFailure`, `InstructionsLoaded`) that do not exist in Claude Code's documented hook system and will never fire. It also uses two hook-entry fields (`statusMessage`, `async`) that are not part of the documented hook schema, meaning the `async: true` hooks almost certainly run **synchronously** (or are silently ignored) rather than asynchronously as the plugin author intended.

## Findings

### 1. Plugin manifest / marketplace schema
1. **`plugin.json` (`.claude-plugin/plugin.json`)** — required: `name`. Optional: `description`, `version`, `author`, `homepage`, `license`, `keywords`, and explicit component pointers (`commands`, `agents`, `hooks`, `mcpServers`) that override the default directory-convention discovery. [Docs: docs.claude.com/en/docs/claude-code/plugins]
2. **Directory conventions** — a plugin root is auto-scanned for `commands/` (slash commands as `.md` files), `agents/` (subagents as `.md` files with YAML frontmatter), `skills/` (each skill in its own folder with a `SKILL.md`), and `hooks/hooks.json` (or a `hooks` block inline in `plugin.json`) for hook registrations. This part of cc10x's layout (agents/, skills/, hooks/, commands/) matches documented convention. [Docs: docs.claude.com/en/docs/claude-code/plugins]
3. **Marketplace manifest (`.claude-plugin/marketplace.json`)** — requires `name`, `owner`, and a `plugins` array where each entry has `name` + `source` (git repo, path, or npm-style source). This is a separate schema from `plugin.json` and is not being audited here since the question is scoped to hooks/agents/skills — flagged as out of scope unless a follow-up specifically requests it.

### 2. Hook events — the real documented set
The current documented Claude Code hook lifecycle events are: **`PreToolUse`, `PostToolUse`, `Notification`, `UserPromptSubmit`, `Stop`, `SubagentStop`, `PreCompact`, `SessionStart`, `SessionEnd`.** [Docs: docs.claude.com/en/docs/claude-code/hooks — "Hook Events" reference table] There is no `PostCompact`, `TaskCompleted`, `StopFailure`, or `InstructionsLoaded` event anywhere in this list.

Compaction specifically only exposes **`PreCompact`** as a hook point. Post-compaction context restoration is handled by firing **`SessionStart` with `matcher`/`source` value `"compact"`** — not by a separate `PostCompact` event. cc10x's own `SessionStart` registration (`matcher: startup|resume|compact`) is actually the *correct* mechanism for this, which makes its separate, non-functional `PostCompact` hook entry redundant dead code on top of being invented.

### 4. `statusMessage` and `async` hook-entry fields
4. **`async` is not a documented field on a hook entry.** The documented hook-entry schema is `{ "type": "command", "command": "...", "timeout": <seconds> }`. All command hooks are invoked synchronously by the Claude Code runtime and their process is awaited up to `timeout`; there is no first-class fire-and-forget mode. An unrecognized `async` key is most plausibly silently ignored by the JSON schema loader (extra/unknown keys are commonly tolerated but non-functional), meaning cc10x's two `async:true` hooks (`StopFailure`, `InstructionsLoaded`) — which are *also* invented event names — are doubly non-functional: wrong event name AND a field with no runtime effect even if the event existed. **NON-STANDARD/INVENTED field.** [Docs: docs.claude.com/en/docs/claude-code/hooks — "Hook Definition" schema]
5. **`statusMessage` is not a documented field on a hook entry.** Same reasoning as above: unknown key, no schema meaning, effectively inert extra data unless silently rejected by strict schema validation (in which case the entire hook entry could fail validation). **NON-STANDARD/INVENTED field.**

### 5. `hooks.json` schema (documented)
6. Top-level shape: `{ "hooks": { "<EventName>": [ { "matcher": "<pattern-or-source>", "hooks": [ { "type": "command", "command": "<shell command>", "timeout": <seconds, default 60> } ] } ] } }`. `type` currently only supports the literal value `"command"` — there is no `"prompt"`, `"async"`, or other type value documented. Matcher semantics differ by event: for `PreToolUse`/`PostToolUse` it's a regex/pipe-list against tool names; for `SessionStart` it's the session source (`startup`, `resume`, `clear`, `compact`); for `PreCompact` it's the trigger (`manual`, `auto`); events like `Stop`, `SubagentStop`, `Notification`, `UserPromptSubmit`, `SessionEnd` typically have no meaningful sub-matcher and are registered with an empty/omitted matcher, which is what cc10x does for `SubagentStop`, `PreCompact`, and `Stop` — correct usage where the event itself is real. [Docs: docs.claude.com/en/docs/claude-code/hooks]

### 6. Exit codes and stdout/stderr handling
7. **Exit code 0** = success. For most hooks, `stdout` is only surfaced to the user in transcript/verbose mode; for `UserPromptSubmit` and `SessionStart`, stdout content is explicitly injected into the model's context (this is the real mechanism cc10x should use for "loading instructions," rather than the invented `InstructionsLoaded` event). [Docs: docs.claude.com/en/docs/claude-code/hooks — "Hook Output" section]
8. **Exit code 2** = blocking error; `stderr` is fed back to Claude and the triggering action is blocked/reconsidered where blocking is meaningful (e.g., `PreToolUse` cancels the tool call, `UserPromptSubmit` blocks the prompt, `Stop`/`SubagentStop` forces continuation instead of stopping). For events where there's no in-flight action to block (`SessionStart`, `SessionEnd`, `Notification`, `PreCompact`, `PostToolUse` since the tool already ran), exit 2 still surfaces `stderr` to Claude/user but cannot "undo" anything.
9. **Other non-zero exit codes** (e.g., 1) = non-blocking error; `stderr` is shown to the user only, and Claude Code continues execution normally.

### 7. Subagent (`agents/*.md`) frontmatter
10. Documented/real fields: **`name`** (required, unique identifier used by the Task tool), **`description`** (required, natural-language text the orchestrating model uses to decide when to delegate — this is the auto-invocation trigger), **`tools`** (optional comma-separated allow-list; omitting it means the subagent inherits the full default tool set), **`model`** (optional — `sonnet`/`opus`/`haiku`/`inherit`). [Docs: docs.claude.com/en/docs/claude-code/sub-agents]
11. **`color`** — used in some example configs/UI theming for subagent chat display; treat as a soft/cosmetic field rather than a functionally load-bearing one. Low confidence without live doc confirmation; flagged as an assumption.
12. **No documented "thinking level" / "effort" frontmatter field exists** for subagents. Extended-thinking behavior is controlled via the `model` selection and runtime/CLI flags, not a dedicated YAML key in agent frontmatter, to the best of available knowledge. If cc10x's agent files declare such a field, treat it as inert metadata.
13. **Subagent invocation mechanism**: subagents are invoked exclusively via the **Task tool** — the orchestrating model matches a delegation intent against subagent `description` text (or explicit `subagent_type` naming) and dispatches a Task tool call; there is no separate deterministic dispatcher.
14. **Context isolation**: each subagent invoked via the Task tool automatically starts with a **fresh, separate context window** — this isolation is a structural property of the Task-tool subagent mechanism itself, not something toggled on/off by a frontmatter field like a "skills" list. There is no documented frontmatter key that controls "inherit no context" — omitting fields does not create isolation; isolation is the default and only mode for Task-tool subagents. Any cc10x documentation implying a frontmatter switch controls this is describing a **prompting/architectural convention layered on top of an already-isolated mechanism**, not a distinct guaranteed API contract. This claim carries **medium confidence** absent live doc re-verification.

### 8. Skills (`SKILL.md`) frontmatter and auto-invocation
15. Real documented frontmatter fields: **`name`**, **`description`** (both required — description is the single signal used for matching), and an **`allowed-tools`**-style restriction field for scoping tool access within the skill (naming may be `allowed-tools` rather than a bare `tools` key — flagged as needing live re-verification since the exact key name is easy to misremember). [Docs: docs.claude.com/en/docs/claude-code/skills]
16. **Auto-invocation is genuinely non-deterministic.** Claude Code loads all installed skills' `name`/`description` metadata into context and the **model itself** semantically judges relevance to decide whether to invoke a given skill (functionally similar to how subagent delegation works) — there is no deterministic keyword/regex router underneath. This directly matters for cc10x's router-driven dispatch design: **routing reliability is bounded by the model's judgment quality on skill descriptions each turn, not by a guaranteed mechanical trigger.** This is a documented characteristic of the feature, not an implementation bug — but it means any cc10x claims of deterministic/guaranteed skill dispatch overstate what the platform actually guarantees.

## Overall verdict
**The cc10x plugin is NOT using real, current Claude Code APIs throughout its hooks configuration.** Of the 10 registered hook events, 6 are real (`PreToolUse`, `SessionStart`, `PostToolUse`, `SubagentStop`, `PreCompact`, `Stop`) and 4 are invented/non-standard and will **silently never fire** (`TaskCompleted`, `PostCompact`, `StopFailure`, `InstructionsLoaded`). Additionally, both custom hook-entry fields observed (`statusMessage`, `async`) are non-standard and will have no effect on runtime behavior — meaning the two `async:true` hooks are broken on two independent axes (wrong event name + non-functional field). On the agent/skill side, the platform's context isolation for subagents is structural (not frontmatter-toggled) and skill auto-invocation is fundamentally a probabilistic, model-judged mechanism rather than a deterministic router — both facts that materially undercut any cc10x documentation claiming guaranteed deterministic dispatch behavior. **Recommended fix for cc10x:** remove the 4 dead hook registrations (or replace `TaskCompleted`→`SubagentStop`, `PostCompact`→ delete (already covered by existing `SessionStart` compact matcher), `StopFailure`→ inspect transcript inside the real `Stop` hook, `InstructionsLoaded`→ `SessionStart` with source `startup`), strip `statusMessage`/`async` from all hook entries, and re-validate agent/skill frontmatter fields against a live fetch of current docs.
