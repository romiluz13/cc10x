### Issues Found

**CRITICAL — Four hook events in `hooks.json` may not exist in real Claude Code, silently disabling enforcement.** `plugins/cc10x/hooks/hooks.json` declares `TaskCompleted`, `PostCompact`, `StopFailure`, and `InstructionsLoaded` hook events (also claimed as "Claude Code-native hooks" in `README.md:538-549`). The documented/known Claude Code hook event set is `PreToolUse`, `PostToolUse`, `Notification`, `UserPromptSubmit`, `Stop`, `SubagentStop`, `PreCompact`, `SessionStart`, `SessionEnd` — notably `SessionEnd` (a real event) is absent from `hooks.json` entirely, while these four unfamiliar names are present. If Claude Code silently ignores unregistered hook-event keys in a plugin's `hooks.json` (typical behavior for unknown keys in most host configs), then: the circuit-breaker hook backstop (`cc10x_task_completed_guard.py` `check_circuit_breaker`, claim #10) and the task-metadata audit (claim #12 enforcement) never execute in a real install — the router's own LLM-side self-counting becomes the ONLY enforcement, exactly the single point of failure the hook was built to eliminate. Per this project's own AGENTS.md rule ("Any time the project pulls in an external technology... you MUST validate latest-version APIs before writing code that uses it"), there is no evidence in the repo that these four event names were validated against current Claude Code hook documentation before being shipped. Also related: the entire `TaskCreate`/`TaskList`/`TaskGet`/`TaskUpdate` task-primitive API this router depends on for resume/hydration (§4) is not a documented native Claude Code tool set either — though the router at least has an explicit fallback for that specific gap (§12 inline-fallback triggers on "TaskCreate/TaskList are absent"), there is no equivalent fallback or self-check for "did my hook actually register/fire."

**HIGH — Stale internal tooling references across docs, tests, and hooks/README point to deleted script paths.** The v11.1.0→HEAD refactor moved `cc10x_harness_audit.py`, `cc10x_workflow_replay_check.py`, `cc10x_latency_audit.py`, `cc10x_doc_consistency_check.py`, `cc10x_review_package.py`, `cc10x_phase_brief.py`, `cc10x_worldclass_benchmark.py`, `cc10x_live_harness_runner.py` out of `plugins/cc10x/scripts/` into `plugins/cc10x/tools/` (de-prefixed). Confirmed by running `python3 plugins/cc10x/scripts/cc10x_harness_audit.py` → `No such file or directory`. But `plugins/cc10x/tests/live/manifests/cc10x-bootstrap.json:19,28,37,46` and `plugins/cc10x/hooks/README.md:36` still reference the old `plugins/cc10x/scripts/cc10x_harness_audit.py` / `cc10x_workflow_replay_check.py` / `cc10x_latency_audit.py` / `cc10x_doc_consistency_check.py` paths, which will fail on any run of the "live bootstrap" test manifest. Multiple `docs/*.md` files (`cc10x-orchestration-safety.md`, `prompt-change-checklist.md`, `cc10x-orchestration-bible.md`, `2026-06-17-HANDOFF.md`) also cite the old paths. This is outside the strict router-kernel file list but is directly responsive to the task's instruction to trace where removed scripts went — it's a real, currently-broken reference, not stale prose.

**MEDIUM — `PostToolUse` "block" language slightly overclaims mechanism.** `SKILL.md` §2a and the guard script comment (`cc10x_posttooluse_artifact_guard.py:150-156`) describe exit code 2 as "rejects... a write" / blocking the write. In real Claude Code PostToolUse semantics, the tool call has already completed by the time the hook runs — exit 2 surfaces corrective feedback to the model (which is accurately what the code does; it tells the model to rewrite the artifact) but cannot actually prevent the malformed file from having been written to disk. The SKILL.md prose ("the guard rejects... a write") could mislead a reader into thinking the write itself is prevented rather than flagged-for-correction after the fact. Low practical risk since the router is instructed to do its own read-back gate first (§6 "Read-back gate (MANDATORY)... the router must not rely on the guard alone").

**LOW — `SessionStart` matcher `"startup|resume|compact"` may not match real SessionStart `source` values.** I'm not fully certain `compact` is a valid `source` value for the `SessionStart` event (I recall `startup`, `resume`, `clear` as the typical set) — combined with the separate, likely-nonstandard `PostCompact` event above, this looks like the authors may have modeled two different mechanisms for the same "context came back after compaction" concern without confirming which one (if either) actually fires. Flagging as LOW confidence/LOW severity since I cannot fully verify the current real value set from within this sandbox.

**LOW — memory finalization rubric terminology drift.** See item 13 above: `SKILL.md` §13 doesn't literally use "KEEP/SUMMARIZE/DROP" (that phrase lives in the global AGENTS.md `/compact-safe` skill description, not the router). Functionally equivalent (persist learnings, prune completed items, defer non-blocking findings) but the specific rubric name from the keep-inventory claim isn't textually present in the router kernel.

**Note — scripts/ consolidation is clean.** `cc10x_instructions_loaded_audit.py`, `cc10x_postcompact_context.py`, `cc10x_stop_failure_log.py`, `cc10x_subagent_stop_audit.py` were correctly folded into `cc10x_event_logger.py` (argv-dispatched subcommands, verified each branch present and logically equivalent to the original name). `cc10x_precompact_state.py` + `cc10x_stop_persist.py` correctly merged into `cc10x_state_persist.py` (argv-dispatched `precompact`/`stop`). `cc10x_reference_benchmark.py` was deleted outright — this matches the plan doc's note that it was a "100% bloat (duplicate)" of `cc10x_worldclass_benchmark.py`, so deletion (not migration) is the correct outcome, not a regression.

**Note — PreToolUse (git/memory guards), PostToolUse, SessionStart hook implementations are internally solid.** `cc10x_git_guard.py` correctly implements the single-use, TTL-bound, operation-scoped approval-token flow matching the prose in `workflow-artifact-and-hook-policy.md` exactly. `cc10x_pretooluse_guard.py` and `cc10x_hooklib.py::pretool_deny`/`session_context` use the correct, real `hookSpecificOutput.permissionDecision`/`additionalContext` JSON protocol for `PreToolUse`/`SessionStart` — this is accurate to documented Claude Code hook JSON output and gives me confidence the CRITICAL finding above is about specific event *names*, not a wholesale misunderstanding of hook mechanics.

### Verdict

The router kernel's orchestration *logic* (routing table, complexity gradient, dispatcher, contracts, remediation/circuit-breaker rules, re-review gate, memory finalization) is coherent, internally consistent, and all 13 claimed innovations are genuinely present and wired in the current code — not just described in prose — with only minor terminology drift on item 13. The v11.1.0→HEAD script consolidation is legitimate and traceable for every renamed/merged file. However, this is **not publish-ready** until two things are resolved: (1) the four non-standard hook event names (`TaskCompleted`, `PostCompact`, `StopFailure`, `InstructionsLoaded`) are validated against current, real Claude Code plugin hook documentation — if they don't fire, the project's two hook-enforced "backstop" claims (circuit breaker, task-metadata audit) silently degrade to prose-only enforcement, which is exactly the failure mode FLAW-003's fix was designed to close for the artifact guard; and (2) the stale `plugins/cc10x/scripts/cc10x_*` references in `hooks/README.md` and the `cc10x-bootstrap.json` live-test manifest are updated to the new `tools/` paths, since those are currently broken (verified by direct execution). Neither issue touches the router's orchestration reasoning itself, but both undermine confidence in the project's own claimed safety net.
